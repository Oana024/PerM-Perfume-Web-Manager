1. Introduction

1.1 Purpose

Perfume-Web-Manager este o resursa web care se ocupa de managementul resurselor dintr-o parfumerie.

1.2 Document Conventions

Documentul este scris in formatul Scholarly descris la adresa: https://w3c.github.io/scholarly-html/. Documentul este codificat in UTF-8 si este afisat intr-un fisier html.

1.3 Intended Audience and Reading Suggestions

Acest document este destinat developerilor, project managerilor, staff-ului de marketing, utilizatorilor, echipei de testare. Documentul contine o descriere a aplicatiei si a modului de utilizare, fiind organizata pe mai multe puncte principale, care contin subpuncte mai mici care explica diverse parti ale aplicatiei. Este recomandat ca documentul sa fie citit in in ordinea in care sunt numerotate subpunctele, incepand cu introducerea.

1.4 Product Scope

In cadrul aplicatiei utilizatorii isi pot crea un cont, iar pe baza profilului ales le vor fi recomandate diverse parfumuri. In plus, parfumurilor recomandate le pot fi aplicate filtre suplimentare din meniul "Filters" pentru o selectie cat mai avansata. Fiecare produs va avea detalii referitoare la ingredientele utilizate, pret, tag-uri asociate care vor duce catre produse care au aceleasi specificatii si relatii cu alte parfumuri inrudite. Totodata de pe pagina unui produs clientul isi poate comanda un parfum apasand butonul "Order" si completand cu datele necesare. In partea de jos a paginii produsului este sectiunea de comentarii, unde clientul poate vedea ce au spus altii despre acel produs si poate adauga si el la randul lui un review acelui produs.

1.5 References

O documentatie a REST API - ului folosit a fost generata utilizand swagger si poate fi consultata la adresa: http://localhost/perM-Perfume-Web-Manager/API/documentation/.

2. Overall Description

2.1 Product Perspective

Aplicatia este una de tip e-commerce, care ajuta utilizatorii sa-si aleaga parfumuri dupa diverse criterii alese de ei. Fiecare produs are asociate comentariile altor utilizatori referitoare la acel produs, astfel clientul poate vedea opinii in legatura cu produsul pe care intentioneaza sa-l achizitioneze.

2.2 Product Functions

  • Creare/accesare cont
  • Filtrare produse
  • Acesarea paginii produsului
  • Comanda produsul

2.3 User Classes and Characteristics

Aplicatia are la 4 clase:

  • User: reprezinta utilizatorul, are ca atribute numele, prenumele, data nasterii, adresa de e-mail, username si alte date ajutatoare pentru creearea profilului utilizatorului. Ca functionalitati se regasesc crearea/stergerea unui utilizator, actualizarea atributelor, afisarea atributelor tuturor utilizatorilor sau a unuia specific.
  • Product: reprezinta produsul, are ca atribute numele, pretul, brand-ul si atribute specifice de exemplu ocazia, aroma. Ca functionalitati se regasesc crearea/stergerea unui produs, actualizarea atributelor unui produs si afisarea tuturor produselor sau al unui specific.

2.4 Operating Environment

Aplicatia este un site web, accesibil utilizatorilor accesand browser-ul, de exemplu Google Chrome, Microsoft Edge, Mozilla Firefox etc. Aplicatia este menita sa functioneze atat in format desktop pe orice sistem de operare, cat si mobile, avand design responsive.

2.5 Design and Implementation Constraints

Aplicatia foloseste pentru stocarea datelor MySQL. La baza aplicatia are un REST API care se ocupa cu adaugarea/stergerea/actualizarea/citirea din baza de date. API-ul este scris in limbajul PHP. Comunicarea intre partea de backend si frontend se realizeaza utilizand to limbajul PHP.

2.6 User Documentation

Pentru a utiliza aplicatia utilizatorul trebuie sa-si crezee un cont. Acest lucru poate fi facut apasand butonul Sign up si completand informatiile necesare. De fiecare data cand utilizatorul vrea sa-si aceseze contul va apasa pe butonul de Login si va completa cu username-ul si parola. Pentru a achizitiona un produs utilizatorul va avea optiunea de a cauta folosind filtrele pentru rezultate cat mai potrivite. Informatiile despre un produs vor putea fi consultate pe pagina produsului avand si comentarii de la ati utilizatori sau alte parfumuri inrudite cu cel ales. Cand utilizatoul s-a hotarat asupra unui produs, acesta trebuie sa apase butonul "Order" de pe pagina produsului (trebuie sa fie logat pe site) si sa completeze datele necesare si sa apese "Submit".

3. External Interface Requirements

3.1 User Interfaces

Legatura vizuala dintre aplicatie si utilizator se realizeaza pe mai multe pagini ale aplicatiei.

  • Pagina de Sign-Up contine un formular compus din mai multe campuri care specifica informatia ce trebuie completata, alaturi de campurile gender si favourite taste care trebuie ales dintre optiunile listate. In partea de jos este butonul de "Sign-up" care va fi apasat dupa ce toate informatiile au fost introduse.
  • Pagina de Login contine un formular cu doar 2 campuri, unul pentru username si altul pentru parola si butonul de Login care va fi apasat dupa completarea campurilor.
  • Pagina principala contine in partea stanga o zona unde sunt butoanele folosite pentru filtrarea produselor dupa preferinte (ocazie, anotimp, brand). In resul paginii se regasesc produsele care apar ca poza produsului, numele produsului si pretul.
  • Pagina produsului contine in partea de sus informatii despre produs. Poza in partea stanga, iar in partea dreapta numele, brand-ul, pretul, descrierea produsului si butonul de order care va aparea doar atunci cand utilizatorul este conectat. Mai jos se gasesc detaliile produsului si tag-urile asociate. In partea de jos apar produse inrudite cu cel ales, iar mai jos sunt comentariile lasate de alti utilizatori si optiunea de a lasa un comentariu atunci cand clientul este conectat.
  • Pagina de about, unde utilizatorul poate gasi o descriere a aplicatiei.
  • Pagina de help care contine intrebari frecvente si raspunsul la aceste intrebari.

3.2 Hardware Interfaces

Cand utilizatorul apasa butonul "Sign up" este redirectionat pe pagina unde se gaseste formularul de inregistrare. Dupa ce completeaza formularul informatia este verificata sa fie corecta (formaul datei de nastere, al adresei de email) si se interogeaza baza de date pentru a verifica sa nu fie deja un utilizator cu acel username sau adresa de email. Se vor afisa mesaje specifice de eroare, iar in caz de succes utilizatorul este redirectionat catre pagina principala. Cand utilizatorul apasa butonul "Login" este redirectionat pe pagina unde se gaseste formularul de login. Utilizatoul trebuie sa introduca username-ul si parola folosite atunci cand si-a creat contul. Daca este vreo problema (username-ul nu este in baza de date sau parola este gresita) se va afisa un mesaj specific, iar daca informatiile sunt corecte utilizatorul va fi trimis pe pagina principala. Cand utilizatorul plaseaza o comanda, stock-ul se modifica in baza de date, iar de fiecare data cand acesta este afisat se verifica in baza de date daca mai sunt produse pe stock in caz contrat se afiseaza "UNAVAILABLE" in locul butonului de "Order".

3.3 Software Interfaces

Aplicatia comunica cu o baza de date MySQL in care sunt stocati utilizatorii, produsele, comentariile si comenzile plasate. Aplicatia interactioneaza cu tabelul users din baza de date in momentul in care un utilizator vrea sa-si creeze un cont sau vrea sa se logheze intr-un cont deja existent, cat si atunci cand sunt afisate produsele pe pagina principala, fiind afisate cu prioritate cele care corespund aromei preferate a clientului. Tabela products este interogata in momentul in care se afiseaza produsele pe pagina principala sau se da click pe un anumit produs si trebuie afisate informatiile specifice acestuia, cat si in momentul in care un produs este comandat pentru a putea modifica stocul existent. Cand utilizatoul adauga un comentariu unui produs acesta este adaugat in baza de date si afisat in josul paginii impreuna cu restul comenatriilor de la acel produs. Cand o comanda a fost plasata cu succes se adauga in baza de date informatii despre aceasta care vor fi folosite pentru a vedea niste statistici ale vanzarilor. Pentru a crea rapoarte de stocuri, vanzari in format pdf a fost folosita libraria fpdf.

3.4 Communications Interfaces

Aplicatia comunica cu browser-ul web al clientului utilizand standardul HTTP.

4. System Features

4.1 Sign-up/Login System

Sistemul de Sign-up sta la baza aplicatiei. Fara sa fie autentificat utilizatorul poate doar sa vada produsele, fara a putea comanda. Pentru a se inregistra utilizatorul trebuie sa completeze urmatoarele date: nume, prenume, data nasterii, email, username, parola, gender si aroma preferata de parfum. Sistemul se asigura ca aceste date respecta formatul cerut al datei de nastere, al email-ului, in caz contrar utilizatorul este atentionat ca a gresit. Sistemul verifica si ca adresa de email, respectiv username-ul sa fie unice, la fel atentionand-ul in cazul in care exista deja un cont cu acele informatii. Aceasta verificare se intampla cu ajutorul REST API-ului care se ocupa de crearea contului. Informatiile sunt preluate si transmise catre API, care va trimite inapoi un raspuns in functie de corectitudinea datelor. Sistemul de login functioneaza in mod asemanator, de data acesta verificandu-se ca utilizatorul sa aiba deja cont pe site si informatiile introduse sa corespunda. Unui utilizator unic ii este atribuita o singura parola (aleasa de el la crearea contului).

4.2 Order System

Cand utilizatorul gaseste un produs care ii place si doreste sa-l achizitioneze trebuie doar sa apese butonul de "Order" care apare doar in momentul in care utilizatorul este inregistrat si produsul este in stoc. Apasarea butonului "Order" va deschide un formular de comanda, care trebuie completat cu nume, prenume, numar de telefon, adresa de email si adresa unde sa ajunga produsul. Dupa ce utilizatorul introduce toate aceste informatii si apasa butonul "Submit", in baza de date se vor adauga informatii despre comanda care vor distribuite mai departe catre firma de curierat care se va ocupa ca produsul sa ajunge la client.

5. Other Nonfunctional Requirements

5.1 Performance Requirements

5.2 Safety Requirements

O mentiune importanta legata de securitate este aceea ca utilizatorul sa foloseasca o parola cat mai complexa si de preferat nefolosita si in alte parti. Cu cat parola este mai complexa cu atat este mai greu de spart, iar in cazul nefericit in care cineva a reusit sa afle parola daca este folosita si pe alte site-uri, cu ajutorul email-ului si al parolei pot avea loc evenimente neplacute. Astfel recomandarea este ca parola sa fie cat mai complexa si unica fata de alte parole folosite pe alte site-uri.

5.3 Security Requirements

Legat de securitatea aplicatiei s-au luat masuri cum ar fi criptarea hash a parolei atunci cand este salvata, astfel daca ajunge sa aiba acces la baza de date cine nu trebuie, nu ii va fi usor sa afle parola aceasta fiind criptata. O alta masura de securitate este ca user-ul sa fie autentificat atunci cand face o comanda pentru a fi prevenite abuzurile, iar in cazul in care se face abuz de aplicatie cand utilizatorul este autentificat, se pot colecta date si contul poate fi blocat sau sters.

5.4 Software Quality Attributes

Un atribut al aplicatiei este acela ca prin intermediul design-ului responsive site-ul poate fi accesat la fel de usor si de pe desktop si de pe mobile. Aplicatia este simpla si usor de intretinut, anumite parti din aplicatie (login, sign-up) putand fi folosite si in cadrul altor aplicatii.

5.5 Business Rules

Aplicatia ar putea dispune de un angajat care sa cunoasca foarte bine aplicatia si care sa poata raspunde clientilor care intampina probleme a caror rezolvare nu se gaseste in sectiunea HELP.
